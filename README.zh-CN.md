# Vocabulary Test UI

[English](/README.md) | [简体中文](/README.zh-CN.md)

这是一个基于 Next.js（App Router）的“英文词汇量/词汇水平估计”测试前端 Demo。  
目标：做成一个既能体现**产品设计**（测量逻辑、迭代路线、输出口径清晰），又能体现**工程落地**（数据结构/算法可扩展、代码边界清晰）的作品，用于面试展示与长期备份维护。

## 演示入口
- 本地运行后访问： http://localhost:3000/test

## 当前 Demo 做了什么
- 以“句子”为单位的测试流程（更接近真实理解场景）
- 支持“按词提示/辅助理解”（例如点击单词弹出提示），但不阻塞做题流程
- 结果页输出（当前为 demo 级别的口径 + UX 基础搭建）

## 为什么用句子而不是纯单词表（产品视角）
纯单词表容易被刷题、信号质量弱；句子交互更接近真实阅读理解，有利于：
- 更可靠地捕捉“认识/半认识/不认识”的不确定性
- 为后续“自适应测试”提供基础：用更少题数达到同等置信度

## 当前范围（v0）
- 固定的小题库（demo 规模）
- 已完成的 UX 优化：
  - 移除频繁弹出的“理解校验”干扰
  - 提示气泡位置优化，避免遮挡句子
  - 翻译/提示出现时，“下一句”依然可用（允许跳过）
- 结果页口径：以“误差/错误”作为表达基础（而不是模糊的“可信度”措辞）

## 下一阶段（Cursor 路线 / MVP 规划）
### 1）数据与结构
- 定义句子条目数据结构（建议字段）：
  - id、sentence、targetWords、lemma/wordFamilyId、词频档位、难度、元数据
- 按词频分层组织题库（高频→中频→低频），支持动态抽题

### 2）自适应选题（MVP）
- 维护“词汇水平估计值 + 不确定性”
- 每题选择尽量“最有信息量”的句子：
  - 优先选择接近当前估计水平的题（太简单/太难信息量低）
- 满足“误差足够小”或“到达最大题数”时停止

### 3）输出（MVP）
- 输出基于 word family 的词汇量估计（带误差区间）
- 不确定性较大时给出建议：例如“再测 N 题可把误差降到 X”

### 4）工程化与可维护性
- 分离关注点：
  - /data：题库、词频分层、word family 映射
  - /lib：选题/计分/停测规则
  - /app：UI 状态机与页面
- 加最小的埋点接口（先本地）：题数、错误、耗时、提示/跳过使用情况

## 建议的目录结构
- app/            Next.js 路由（测试流程、结果页）
- components/     UI 组件
- lib/            选题与计分逻辑
- data/           题库与资源（后续新增）
